- 작성일: 2026-07-08
- 근거: `gatherings-gap-analysis-2026-07-08.md` (갭 21건 실측 분석)
- 목표 한 줄: **"공지→투표→확정→모집→당일→정산·후기"의 모임 라이프사이클에서 사람 손과 사고 가능성을 제거한다.**

---

## 프로젝트 비전

현재 모임 기능은 개별 조각(투표·RSVP·리마인더·정산)이 대부분 갖춰졌으나, **조각 사이의 연결 고리**(취소 알림, 대기자 승격, 확정 즉시성, 실참석 검증)가 비어 있어 운영진 수작업과 신뢰 사고 위험이 남아 있다. 본 프로젝트는 신규 기능 추가보다 **루프 완결**에 집중한다.

## 페이즈 구성 (4단계 + 합류 트랙)

### Phase 1 — 신뢰·정확성 (즉시 착수 권장 · 총 규모 S~M · 예상 1게이트)
| 항목 | 갭 | 규모 | 내용 |
|---|---|---|---|
| 동반인 입력 UI | G6 | S | 정산·정원에 이미 반영되는 `companions`의 입력구 신설 — 구조적 과소집계 버그 해소 (최고 ROI) |
| 취소·연기 알림 | G1 | S~M | status→cancelled·일시 변경 시 참석/미정 신청자 인앱 알림 — 현장 사고 방지 |
| 투표 확정 정합 | G10·G11·G12 | S~M | 마감 후 "확정 대기" 안내, auto+마감일 없음 경고(폼 필수화), 자유텍스트 시간대 확정 시 운영진 시각 확인 |

**성공 기준**: 취소 모임에 알림 발송 로그 존재 / companions>0 데이터 발생 / "왜 확정 안 되냐" 문의 0건.

### Phase 2 — 모집·참여 완결 (총 규모 M · 예상 1~2게이트)
| 항목 | 갭 | 규모 | 내용 |
|---|---|---|---|
| 대기자(waitlist) | G2 | M | 정원 초과 시 waitlisted 등록, 취소 발생 시 1순위 자동 승격+알림 |
| 체크인·노쇼 | G3 | M | 콘솔 명단 체크인 토글(attended), 잔디 +5점을 실참석 기준으로 전환 |
| 회비 미납 흐름 | G5 | M | 미납 due D-day 알림 + 마이페이지 "내 미납 회비" 집계 |
| 투표 개시 알림 | G14 | S | poll 신규 개설 시 회원 알림(초기 응답률↑) |

**성공 기준**: 취소→승격 자동 처리 1건 이상 / streak 적립이 attended 기준 / 미납 알림 발송.

### Phase 3 — 운영 효율·인사이트 (총 규모 M~L · 예상 2게이트)
| 항목 | 갭 | 규모 | 내용 |
|---|---|---|---|
| 운영 통계 대시보드 | G18 | M | 참석률 추이·반복 참석자·유형별 참여·회비 회수율·노쇼율 (Phase 2의 attended 데이터 활용) |
| 회비 자동화 | G19 | M | attending 확정 시 due 자동 생성 옵션 + 일괄 납부 처리 |
| 반복 모임 | G4 | M | "이 모임 복제" 최소 구현 → 반복 규칙은 후속 |
| RSVP 정정·철회 | G7·G8 | M | 게스트 신청 확인·취소 경로(토큰), 회원 신청 철회 |

### Phase 4 — 프라이버시·확장 (아키텍처 결정 선행 · L)
| 항목 | 갭 | 규모 | 내용 |
|---|---|---|---|
| 비공개 메타 분리 | G15 | M~L | invitedUserIds·본문의 공개 read 잔존 해소 — rules·목록 증명 재설계 필요(**운영진/아키텍처 결정 선행**) |
| 레거시 토큰 백필 | G16 | S | 기존 private 이벤트 shareToken 필드 제거 백필 1회 |
| 비공개 poll 공유 | G17 | M | 토큰 라우트 기반 비공개 종합 공유 |
| 소소한 견고성 | G20·G21 | S | 게스트 정원 트랜잭션화, poll 폴러 통합 |

### 합류 트랙 (별도 진행 중)
- **G9 게스트→회원 전환 퍼널**: "비로그인 투표(이름+학번)+잠재회원 연계" 트랙이 구현 중 — 완료 시 Phase 2의 게스트 흐름(G7)과 접점 정리.

## 외부 의존·운영진 결정 필요
1. **결제 연동 여부** — 현행 오프라인 수동 회비는 명시적 설계 결정. 전환하려면 PG·계좌 정책 결정 필요 (Phase 3 회비 자동화는 결제 없이도 가치 있음).
2. **G15 비공개 메타 분리 방식** — 공개 목록 쿼리 증명과 결합된 rules 재설계라 아키텍처 선택지(별도 컬렉션 vs 서버 조회 전환) 결정 필요.
3. **노쇼 페널티 정책** — 체크인 데이터로 무엇을 할지(경고/제한)는 운영 정책 사안.

## 진행 방식 제안
- Phase 단위 게이트(빌드·배포·QA 스모크) + codex 교차검증은 Phase 1·2 완료 시점 2회.
- Phase 1은 지금 바로 착수 가능(전부 코드로 해결, 외부 의존 없음).
