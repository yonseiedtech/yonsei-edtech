## 0. 총평 — 제품이 지금 서 있는 곳

이 서비스는 **엔지니어링 성숙도가 제품 검증 속도를 크게 앞질렀다.** 8개 라운드(v6~v13)의 자율 고도화로 기능·측정·관측·리허설 인프라가 거의 완비됐고, v13 해커톤 종단 리허설로 "만든 것이 작동한다"까지 실증됐다. 그러나 PM 렌즈로 보면 구조적 불균형이 뚜렷하다:

1. **표면 299개 vs 사용자 수십 명** — 1인당 라우트가 5개를 넘는 규모. 기능 부족이 아니라 **소비·집중·유지비**가 리스크의 중심이다.
2. **백로그가 코드 실측에서만 나온다** — v13 계획서 스스로 "자율 분석 · 대화형 인터뷰 없음 · **코드·git 실측만**"(v13 §헤더)이라 명시. 8라운드 동안 **사용 데이터·사용자 피드백이 기획 입력으로 쓰인 증거가 docs/plans 전수에 없다.** 측정 인프라(visit-tracker·funnel·loyalty)는 쌓이는데 의사결정은 코드 grep으로 이뤄지는 역설.
3. **8월 실전의 임계 경로가 코드에서 운영으로 넘어갔다** — v14 외부 의존 X1~X6(심사위원·루브릭·장소·신학기 콘텐츠·알림 정책)이 전부 미결. 코드는 준비됐는데 **행사를 실제로 성립시키는 결정·콘텐츠가 비어 있다.**

아래 갭들은 이 세 축을 여정별로 분해한 것이다.

## 1. 프레임 1 — 가치 제안 vs 실제 사용 (빌드-소비 갭)

### GAP-01. 표면 총량 관리 부재 — "만들기"만 있고 "접기"가 없다
- **영역/여정**: 전 표면 / 제품 전략
- **갭 설명**: `src/app` page.tsx **299개**(회원 대면 ~168 + 콘솔 123 + admin 8). 8라운드 백로그는 전량 "추가·보정"이며, docs/plans 160여 문서 어디에도 **저사용 표면의 통합·숨김·일몰(sunset) 판단 프로세스가 없다.** `visit-tracker`가 방문을 수집 중임에도(상주 인프라, v13 §0-3) "표면별 방문 리그테이블 → 하위 표면 처분" 루프가 0회 실행. 예: `/studio`, `/agents`, `/ai-forum`, `/labs/qa-wall`, `/exam-schedule-admin`, `/research-model` 등은 어느 여정 문서에도 등장하지 않는 고아 후보 표면인데 실사용 여부를 아무도 모른다.
- **사용자 영향**: 회원 — 발견성 소음(Cmd+K·헤더에 표면이 많을수록 핵심 도구 매몰). 운영진 — 유지비·회귀 표면적 증가(빈 상태 ~213파일, 색상 부채 322파일이 바로 이 총량의 함수). 신규 운영진 — 인수인계 난도 상승.
- **제안**: 월 1회 "표면 리그테이블" 리포트(visit-tracker 집계 재사용, 신규 컬렉션 불필요) → 30일 방문 0~N건 하위 표면을 [통합/nav 제거/유지] 3분류하는 **일몰 정책**을 라운드 규율에 편입. 라운드 계획서에 "추가 N개당 정리 후보 1개" 규칙 명문화.
- **Impact/Effort**: **Impact 상 / Effort S** (집계 뷰 S + 정책은 문서)

### GAP-02. 기능별 채택률(adoption) 무측정 — 만든 루프가 돌아가는지 모른다
- **영역/여정**: 학습 루프(진단·암기카드·읽기 타이머·wrapped·리더보드) / 재학생
- **갭 설명**: 진단→약점→암기카드(SM-2)→재진단 루프, 읽기 타이머, wrapped, kudos, weekly-goal 등 **행동 루프형 기능이 10종 이상 LIVE**인데, 이들의 **활성 사용자 수·재사용률을 보여주는 단일 뷰가 없다.** `admin/insights`·`console/insights`는 방문·funnel·loyalty 스냅샷 중심이고, "진단 완료 N명 중 암기카드로 이어간 사람 M명" 같은 **루프 전환율**은 어디에도 없다. v14가 kudos 리더보드를 "볼륨 N≥50 확인 후"로 조건 걸었듯, 판단 기준이 산발적·수동적.
- **사용자 영향**: 사용자는 무영향이나, **제품은 어떤 루프가 죽어 있는지 모른 채 다음 라운드를 계획**한다. 죽은 루프 위에 고도화가 얹히는 낭비(예: 암기카드 SM-2 고도화가 실사용 3명 위에 있을 수 있음).
- **제안**: `console/insights`에 "루프 헬스" 섹션 1개 — 기존 컬렉션(diagnostic results·flashcard reviews·paper_reading_logs·weekly_goal_records·kudos) count 집계만으로 루프별 [주간 활성 N·전환율] 표. LLM 무의존·신규 컬렉션 불필요.
- **Impact/Effort**: **Impact 상 / Effort S~M**

### GAP-03. 콘텐츠 의존 표면의 공급 계약 부재
- **영역/여정**: 아카이브·온보딩·뉴스레터·카드뉴스 / 신입·전체
- **갭 설명**: 아카이브(개념·용어·연구방법·통계 등 20여 표면)는 시드+검수 큐(통합 검수 큐 LIVE)로 돌지만, **온보딩 가이드(`guideItemsApi`)·뉴스레터·카드뉴스·환영 공지 등 "운영진이 계속 써야 사는" 표면은 발행 캘린더·담당자·주기가 어디에도 정의돼 있지 않다.** v14 X4("신학기 콘텐츠 작성 — 운영진 콘텐츠 발행")가 여러 라운드째 외부 의존 표로만 이월. 콘텐츠 노후 감지(v11-H4 freshness queue)는 있으나 **공급 측 약속(SLA)이 없어 감지해도 채울 사람이 미정.**
- **사용자 영향**: 신입 첫인상이 빈 화면일 위험(07-17 JG-a2와 동일 구조 — 표면은 통합됐으나 콘텐츠 공급 문제는 미해결). 뉴스레터·카드뉴스는 발행 공백 시 "죽은 코너"로 보여 서비스 전체 신뢰 하락.
- **제안**: 콘텐츠 표면별 [담당 직책 · 주기 · 다음 발행일]을 `site_settings` 1문서로 관리 + 시즌 체크리스트(v13-H4 공유화 완료)에 발행 항목 자동 연동. 운영진 결정(담당 배정)은 외부 의존으로 명시하되, **개강 전 1회 "콘텐츠 발행의 날" 일정을 사용자에게 제안**하는 것이 코드보다 효과 큼.
- **Impact/Effort**: **Impact 상 / Effort S** (코드 S + 운영 합의)

### 여정 ① 신입 (발견→가입→승인→온보딩→첫 활동)

### GAP-04. 신입 파이프 "코드 완성·실주행 0회" — 개강 전 최대 미검증 구간
- **영역/여정**: 가입~첫 활동 / 신입
- **갭 설명**: v13-H3로 `cohortKeyOf` 8월 가입 보정·`effectiveSemesterStart` 공용화까지 코드는 완결됐으나(`h3-h5-semester-fix-v13-2026-07-21.md`), **테스트 계정으로 가입 4단계→콘솔 승인→승인 이메일 착지→첫 로그인→첫 활동을 이어 걸어본 이력이 0회**(v14 갭③ 실측 동일). 해커톤은 종단 리허설을 했는데(v13-H1) 개강 유입 파이프는 안 했다. 8월은 가입이 몰리는 달 — 승인 이메일 문구·착지 URL·`NewcomerProgressWidget` 타이밍·첫 활동 CTA 마찰이 전부 미측정.
- **사용자 영향**: 신입의 첫 24시간 경험이 검증 안 된 채 개강을 맞음. 승인 이메일이 스팸함행이거나 착지가 어긋나면 **가입 승인→활성화 전환 자체가 무너진다** (수십 명 커뮤니티에서 신입 5명의 이탈은 코호트의 20%+).
- **제안**: v14-H4 그대로 — 8/18 전 테스트 계정 종단 리허설 + [기대 vs 실제] 감사 문서 + 표현 계층 핫픽스. **해커톤 리허설(v13-H1)에서 검증된 방법론 재사용.**
- **Impact/Effort**: **Impact 최상 / Effort S** — 8월 실전 항목 중 ROI 1위

### GAP-05. "발견→가입" 상단 퍼널이 제품 밖에 있다
- **영역/여정**: 인지·유입 / 신입·잠재회원
- **갭 설명**: 잠재회원 캡처(게스트 투표→potential_members)·`console/potential-members`는 LIVE지만, **신입이 학회를 처음 알게 되는 경로(교수 추천·선배 소개·검색)에서 서비스가 하는 일이 없다.** 네이버 서치어드바이저 인증은 사용자 발급 대기(X7)로 수 라운드째 정지, OG 메타는 정비됐으나(v10-M6) **"가입 전 공개 화면에서 무엇을 보여줘 가입을 설득하는가"의 공개 랜딩 전략이 부재** — `/about`은 기관 소개이지 가치 제안(진단·아카이브·멘토링 미리보기)이 아니다.
- **사용자 영향**: 신입 유입이 전적으로 오프라인 구전에 의존. 개강 시즌 유입 기회(신입생 검색·학과 공지)를 흘려보냄.
- **제안**: (a) X7 해소를 사용자에게 1회 리마인드(발급 5분 작업). (b) 공개 상태의 아카이브 대표 콘텐츠 2~3개 + 진단 맛보기 1문항을 `/about` 또는 랜딩에 노출하는 "가입 전 미리보기" S 작업. 대규모 마케팅 기능은 **수십 명 규모에 과잉이므로 비추천** — 검색 인입 + 미리보기까지만.
- **Impact/Effort**: **Impact 중 / Effort S**

### 여정 ② 재학생 연구자 (진단→학습→논문→졸업)

### GAP-06. 연구 여정의 "다음 한 걸음" 개인화가 규칙 시그널을 다 쓰지 않는다
- **영역/여정**: 진단·학습·논문 / 재학생
- **갭 설명**: 루프 부품은 완비 — 진단·약점→개념·암기카드 전타입·설계→논문 import·주간 목표·digest 학습 제안(07-17 스프린트 반영 완료). 남은 갭은 **부품 간 우선순위 중재**: 대시보드에 위젯·배너·넛지(NextActionBanner·NewcomerProgressWidget·SemesterKickoffBanner·주간목표·digest)가 각자 "다음 행동"을 말해 **신호가 병렬 경합**한다. 개인의 단계(진단 미완/약점 학습 중/설계 중/집필 중/심사 준비)를 판정하는 데이터는 이미 다 있는데(진단 결과·설계 문서·주간 목표·읽기 로그) 이를 묶는 **단일 우선순위 규칙**이 없다.
- **사용자 영향**: "무엇부터"의 인지 부하가 사용자에게 전가. 특히 논문학기 재학생은 바빠서 대시보드를 훑고 이탈 — 루프 재진입률 저하.
- **제안**: LLM 없이 규칙 기반 단계 판정 함수 1개(기존 데이터 read-only) → 대시보드 최상단 "지금 단계: OO — 다음 한 걸음" 카드 1장으로 기존 넛지들을 **대표 1개만 승격**(나머지는 접힘). 신규 컬렉션 불필요.
- **Impact/Effort**: **Impact 상 / Effort M**

### GAP-07. 졸업 전환점의 제품 이벤트 부재
- **영역/여정**: 재학생→졸업생 전환 / 재학생·졸업생
- **갭 설명**: 졸업 로드맵(H3)·수료증·wrapped 상시 진입(v12-H4)·포트폴리오 자동적재·PDF는 LIVE. 그러나 **"졸업" 순간에 서비스가 능동적으로 하는 일이 없다** — enrollmentStatus가 졸업으로 바뀌면 학사 위젯이 조용히 사라지고 소비형 알럼나이 위젯으로 교체될 뿐(07-17 JG-c3 구조 잔존). 멘토 토글은 생겼지만 **졸업 시점에 멘토 등록·후배 조언 남기기·포트폴리오 아카이빙을 제안하는 전환 시퀀스가 없다.** 졸업은 리텐션이 꺾이는 유일한 예측 가능 시점인데 무대응.
- **사용자 영향**: 졸업생 이탈이 기본값. 멘토 공급 부족 → 재학생 멘토링 수요 미충족 → 학회의 세대 연결 가치 약화.
- **제안**: 졸업 판정 시 1회성 "졸업 전환 체크리스트" 배너(wrapped 보기·멘토 토글 켜기·학위논문 아카이브 등록·연락처 최신화 — 전부 기존 기능 딥링크 4개). cron 1개 조건 추가 수준.
- **Impact/Effort**: **Impact 상 / Effort S**

### 여정 ③ 졸업생 (멘토링·기여)
