## 1. 코호트 산정 규칙 — `cohortKeyOf(member)`

파일: `src/lib/semester.ts` (기존 `semesterKeyOf` 재사용)

회원의 "가입 학기"를 학기 키(`"YYYY-1"` 전기 / `"YYYY-2"` 후기)로 파생하는 순수 함수. 저장 없음.

우선순위:
1. `enrollmentYear` + `enrollmentHalf`(1|2) → `"${year}-${half}"`. 연락망(`/directory`)의 "입학시점"과 동일 소스라 표기 일관.
2. 위 정보가 없으면 `createdAt`(가입일) → `semesterKeyOf(createdAt)` 로 유도.
3. 둘 다 없으면 `null`(코호트 미상 → 화면 미노출).

`generation`(기수 number)은 학기 키로 매핑되지 않아 산정 키에는 쓰지 않고, 화면에는 참고 배지("N기")로만 병기.

## 2. 온보딩 코호트 화면 — "우리 기수"

- 컴포넌트: `src/features/onboarding/CohortSection.tsx` (신규)
- 진입점: `src/app/steppingstone/onboarding/page.tsx` — 개인 진행률 카드 바로 아래 삽입. (M2가 추가한 `logOnboardingEvent`(ui:onboarding/*) 퍼널 코드는 그대로 보존, 섹션만 추가.)
- 데이터: `useMembers()`(승인 회원만, 500 limit) 재사용 — 신규 네트워크 호출 최소화. 코호트 = `cohortKeyOf(m) === cohortKeyOf(me)`.

구성:
- **헤더**: 학기 라벨(`semesterLabelFromKey`, 예 "2026년 후기 입학") + 동기 수 + `me.generation` 배지.
- **평균 진행률(익명화)**: 온보딩 트랙(`onboardingTrackId`)·항목수(`totalItems`)가 있을 때만. `guideProgressApi.listByTrack`으로 트랙 전체 진행 문서를 받아, 코호트 구성원별 완료율(`완료 항목 수 / totalItems`, 상한 1)을 계산해 **평균 %와 "시작 인원 수"만** 노출. **개인 퍼센트·순위는 미노출** — 개인 식별 비교 없음(평균·분포만).
- **동기 명단**: 같은 코호트 승인 회원(본인 제외)을 이름 + 프로필 링크(`/profile/[id]`) 칩으로 노출. 노출 범위는 기존 회원 디렉토리 수준과 동일(승인 회원·이름·프로필). 없으면 안내 문구.

## 3. 버디 추천 로직

- 기존 `recommendCollaborators(me, candidates, limit)`(`src/lib/collaborator-match.ts`) 재사용.
- 후보를 **같은 코호트 회원으로 한정**해 호출한 뒤, 매칭 근거에 **관심분야 겹침**(`interest_keyword` 또는 `research_topic`)이 있는 매칭만 필터링해 상위 **1~3명** 노출.
  - 이유: 코호트 내에서는 `cohort`/`occupation` 근거가 항상 겹쳐 전원이 추천되므로, 실제 관심분야 겹침이 있는 버디만 남기도록 근거 종류로 필터. 겹치는 후보가 없으면 버디 카드 자체를 미노출.
- 카드: 관심 주제 교집합 칩 + "프로필 보기"(`/profile/[id]`).

## 4. 신입 시퀀스 cron 연동

파일: `src/app/api/cron/semester-start-reminder/route.ts` (신규 cron 없음 — 기존 개강 리마인더에 분기 추가)

  - 응답 JSON에 `onboardingNotifCount` 추가.

## 5. 개인정보 처리

- 명단·버디: `useMembers()`가 반환하는 **승인 회원만**. 이름·프로필 링크는 기존 `/directory`·`/profile/[id]` 노출 수준을 그대로 따름(그보다 넓히지 않음).
- 진행률: **평균·시작 인원 수만** 노출, 개인 퍼센트·순위 미노출(익명화).
- `firestore.rules` 변경 없음. `guide_progress` 조회는 기존 rules 적용. (참고: 코호트 평균 집계는 트랙 단위 `guide_progress` 읽기를 요구하므로, 운영에서 회원의 트랙 단위 조회 권한이 제한적이면 이 평균 블록만 비게 될 수 있음 — 이 경우 명단·버디는 정상 노출되고 평균 카드만 생략됨. rules 확장이 필요하면 별도 검토 항목으로 제안.)
