### 2.2 서론 (Introduction)
- ✅ 도입부의 문제의식이 학술적 호소력 있음
- ⚠️ 학회 차원 디지털 인프라의 **선행 연구 부재**를 명시할 인용 부족

  2. 기여 진술을 "10개 이론의 매핑 매트릭스 + 통합 패턴 3유형(시너지/긴장/보완) 식별 + 후속 실증 연구 6개 트랙 제시" 로 구체화

### 2.3 관련 연구 (Literature Review)
- ✅ 3 sub-section 구조는 적절
- ⚠️ **각 sub-section의 인용 수가 너무 적음** (sub-section 당 2~3편) — 학회지는 sub-section 당 5~8편 권장
- ⚠️ **2.3 이론 통합 연구의 도전** 섹션이 너무 짧음. 본 논문의 방법론적 위치 정당화에 핵심인데 1단락 분량
- ⚠️ **국내 학회 사례 연구 부재** — 국내 학회지 게재라면 한국 사례 인용 필수
- 📝 **수정 요청**:
  1. 각 sub-section을 2~3 단락으로 확장, 인용 5~8편으로 보강
  2. 2.3 섹션을 별도 단락으로 확장하고, 이론 통합 방법론(framework synthesis, integrative review) 인용 추가
  3. 국내 학회·기관의 디지털 인프라 사례 2~3편 검색·인용

### 2.4 방법 (Method)
- ✅ 4가지 자료 원천 + 5단계 분석 절차의 명시는 우수
- ❌ **매핑 강도 평가(●●●/●●/●/◐)의 기준이 모호함** — reviewer 가 가장 의심할 부분
- ❌ **Inter-rater reliability 부재** — 단일 저자 분석이라면 신뢰도 검증 절차 필요
- ⚠️ 운영진 인터뷰 "3~5명 선택적"은 모호. 실시했으면 명시, 안 했으면 향후 작업으로 명확히 표시
- ⚠️ **시스템 코드베이스 분석의 정량 지표 부재** (총 commit 수, 파일 수, LOC 등)
- 📝 **수정 요청**:
  1. 매핑 강도 평가 rubric 명시 — "●●● 핵심 = 디자인 결정 시 해당 이론이 1차 논거로 명시되었고 시스템 핵심 동작에 직접 영향" 같은 정의
  2. Inter-rater reliability 검증 절차 추가 — 본 저자 + 외부 2인이 독립적으로 매핑 후 Cohen's κ 계산
  3. 인터뷰 실시 여부·인터뷰 대상·분석 절차 명확히
  4. 코드베이스 정량 지표를 Appendix A 에 표로 제시

### 2.5 결과 — 10개 이론별 매핑

- 📝 **수정 요청**: 이론별로 다음 구조 권장
  ```

  단락 1: 이론의 핵심 명제 + 주요 인용 2-3편

  단락 3: 본 구현의 한계 + 측정 가능 지표 명시
  ```

## 4. 추가 권장 — 학술적 강도 제고

### 4.1 영문 초록 + 영문 키워드 추가
KAIE·ETR&D 어디든 영문 초록 필수.

### 4.2 매트릭스의 mathematical formalization
매핑 강도를 정량화하여 행 평균(이론별 통합도), 열 평균(도메인별 이론 밀도), 매트릭스 entropy 등 지표 추가. 학술적 정밀성 ↑.

### 4.3 비교 분석 (선택)
다른 학회 사이트(AECT, JSET 등)와의 도메인 비교 표 1개 추가하면 일반화 가능성 논의 강화.

### 4.4 운영 KPI 1개의 사전·사후 비교
어려우면 "Phase 1 도입 전 6개월" vs "Phase 1 도입 후 6개월" 의 활성 회원 비율 비교만 해도 reviewer 만족도 ↑.

### 4.5 "이론적 통합 framework" 의 명시
단순 매트릭스가 아니라 **"본 연구가 제안하는 이론 통합 framework"** 를 도식화하면 contribution 의 학술 단위가 명확해짐.
