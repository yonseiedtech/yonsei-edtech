# 제안서 — 논문 심사·디펜스 운영 보드

- 작성: 2026-07-07 · 근거: codex 서비스 갭 분석(여정 갭 1순위) + 실사용 맥락
- 한 줄 요약: 논문학기생이 **예심→본심→(필요 시)재심사→최종 제출**의 관문을 일정·위원·제출물·결과·수정요구 단위로 추적하고, 운영진이 코호트로 한눈에 관리하는 상태 보드.

## 1. 배경 — 무엇이 비어 있나

논문 여정 도구는 **작성 전 단계가 촘촘**하다(주제 탐색·문헌 매트릭스·모형·계획서·에디터·타이머). 반면 **심사·디펜스 단계는 "연습" 편중**이다.

| 이미 있는 것 | 하는 일 | 한계 |
|---|---|---|
| 디펜스 연습 (`defense_practice_sets`, STT 따라읽기) | 예상 질문 연습 | 실제 심사 일정·결과와 무관 |
| 지도 노트 (`advisor_feedback_notes`) | 지도교수 피드백 반영 추적 | 심사위원 요구사항과 분리, 관문 개념 없음 |
| 종합시험 기록 (`comprehensive_exam_records`) | 종합시험 관문 | 논문 심사와 별개 |
| 여정 5단계 "완성·심사 방어" | 연습·지도노트 CTA만 | "언제 심사인지, 통과했는지, 뭘 고쳐야 하는지"를 담는 곳이 없음 |

**결과적으로 심사 과정 자체는 기억·오프라인 메모·개인 파일에 흩어져 있다.** 특히 우리 전공에서 흔한 **"본심 조건부 통과 → 수정 후 재심사"** 흐름은 수정 요구사항을 항목별로 반영·확인해야 하는데, 이를 관리하는 도구가 전혀 없다.

## 2. 사용자와 문제

- **논문학기생(핵심)**: 예심/본심이 언제인지, 위원이 누구인지, 무엇을 언제까지 제출하는지, 심사에서 나온 수정 요구를 빠짐없이 반영했는지를 한 곳에서 못 본다. 재심사 시 "지난번 지적이 다 반영됐나"를 스스로 증명하기 어렵다.
- **운영진**: 이번 학기 논문학기생이 각각 어느 관문에 있는지, 임박한 심사가 있는지, 서류 마감이 걸린 사람이 있는지를 파악할 도구가 없다(수동 파악).
- **지도교수/자문위원(선택)**: 학생의 수정 반영 현황을 링크로 확인하면 지도 효율이 오른다.

## 3. 핵심 기능 — 개인 심사 보드

관문(Milestone) 단위 상태 카드. 각 관문에 5개 축:

1. **일정** — 심사일·장소, D-day 자동 계산, 개강/학사일정 연동, D-7·D-1 리마인더(기존 크론 재사용)
2. **위원** — 지도교수·심사위원 명단(회원이면 프로필 연결, 외부는 이름·소속 텍스트)
3. **제출물** — 심사본·발표자료·수정대조표 등 체크리스트(+링크). 마감 표시
4. **결과** — 통과 / 조건부 통과(수정 후 재심) / 보류 / 불통과. 판정일 기록
5. **수정 요구 추적** — 심사에서 나온 지적을 항목으로 등록 → 각 항목 `미착수/반영중/반영완료` + 반영 메모. **재심사 전 "N개 중 M개 반영" 게이지** — 이 도구의 심장

기본 관문: `예비발표 → 예심(1차) → 본심(2차) → (조건부 시) 재심사 → 최종 제출·인준`. 전공 실정에 맞게 관문은 콘솔에서 편집 가능.

기존 자산 연계:
- 각 관문에서 **디펜스 연습 세트** 바로 실행(예상 질문 대비)
- 수정 요구 항목을 **지도 노트**와 교차 링크(지도교수 피드백↔심사 요구 통합 뷰)
- 여정 5단계 카드가 이 보드의 요약(다음 관문·D-day·수정 반영률)을 표시

## 4. 운영진 코호트 보드 (`/console/grad-life/thesis-defense` 확장)

현재 "연습 목록"인 콘솔 페이지를 **상태 관리 보드**로 확장:
- 이번 학기 논문학기생 × 관문 매트릭스(누가 어느 단계인지 한눈에)
- 임박 심사·마감 임박 알림, 조건부 통과 후 재심사 대기자 하이라이트
- 관문 정의·기본 제출물 체크리스트 템플릿 관리(기존 템플릿 관리와 통합)

## 5. 데이터 모델 (초안)

```
thesis_reviews/{userId}                 // 학생 1명 = 문서 1개 (본인/staff/지정 지도교수 read)
  milestones: [{
    key, label, scheduledAt, location,
    committee: [{ userId?, name, affiliation, role }],
    deliverables: [{ label, url?, done, dueAt? }],
    outcome: 'pending'|'passed'|'revise'|'hold'|'failed', decidedAt?,
    revisions: [{ text, status:'todo'|'doing'|'done', note?, updatedAt }]
  }]
  currentMilestoneKey, updatedAt
```
- **읽기**: 본인·staff·(옵션)지정 지도교수. **쓰기**: 본인(수정 반영·제출물), staff(관문·결과). → 룰은 최근 정비한 "본인/staff 증명형" 패턴 그대로.
- 인덱스 부담 없음(문서 1개 get, 코호트는 staff가 승인회원 순회 또는 별도 요약 컬렉션).

## 6. 범위와 단계

- **MVP (1차)**: 개인 보드 — 관문 카드 + 수정 요구 추적 게이지 + D-day/리마인더 + 여정 5단계 요약 연동. 마이페이지 연구활동에 탭 또는 여정 5단계 확장으로 편입.
- **2차**: 운영진 코호트 매트릭스 + 콘솔 템플릿 통합.
- **3차(선택)**: 지도교수 열람 링크, 심사 결과 아카이브(졸업 후 후배 참고용, 익명).

## 7. 20명 규모 적합성 · 비범위

- 신규 대형 컬렉션 1개(문서=학생 수)로 가벼움. 알림·리마인더·룰은 전부 기존 인프라 재사용.
- **비범위**: 심사 점수 산정·자동 통과 판정(교수 권한 영역), 외부 심사 시스템 연동, 논문 내용 자체 검증. 이 도구는 **과정 추적**이지 심사 대체가 아니다.

## 8. 성공 지표

- 논문학기생의 심사 보드 사용률(관문 1개 이상 등록)
- 수정 요구 항목 "반영완료" 전환율 — 재심사 준비의 실질 지표
- 운영진의 "임박 심사 사전 인지" 여부(사후 파악 → 사전 파악 전환)

## 9. 다음 결정

- 편입 위치: **마이페이지 연구활동 신규 탭** vs **여정 5단계 인라인 확장** (권장: 5단계 확장 — 여정 맥락 유지)
- 관문 기본값: 우리 전공 실제 흐름(예심·본심 명칭, 재심사 관례)을 운영진이 확정하면 그에 맞춰 시드
- 착수 시점: 8월 개강 준비와 겹치지 않게, C-3 워크숍 이후 별도 사이클 권장
