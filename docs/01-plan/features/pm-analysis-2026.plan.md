## 1. Overview

### 1.1 Purpose

연세교육공학회 웹사이트의 현재 상태를 PM 관점에서 종합 분석하고, 사용자 여정의 pain point를 식별하며, 회원 활성화 및 학회 운영 효율을 높이기 위한 개선 방향과 우선순위를 제안한다.

### 1.2 Background

연세교육공학회 웹사이트는 Next.js 16 + Firebase + Vercel 기반으로 구축된 학술 커뮤니티 플랫폼으로, 주요 기능(회원 관리, 세미나 운영, 게시판, AI 챗봇, 학회보 등)이 이미 구현되어 있다. 그러나 코드 분석 결과, 사용자 여정 전반에 걸쳐 개선 가능한 지점들이 다수 발견되었으며, 현재 부재한 기능들이 학회 운영의 효율성과 회원 참여도를 제한하고 있다.

### 1.3 분석 대상 사용자 유형

- **신규 회원 (신입생)**: 학회 가입 → 온보딩 → 첫 참여
- **기존 회원 (재학생)**: 일상적 학회 활동 참여, 세미나 신청/참여
- **졸업생 (alumni)**: 간헐적 방문, 네트워킹
- **운영진 (staff/president)**: 세미나 운영, 게시판 관리, 회원 승인
- **관리자 (admin)**: 사이트 전반 관리

## 2. 사용자 여정 분석 및 Pain Point

### 2.1 신규 회원 여정

```
발견 → 가입 신청 → 승인 대기 → 로그인 → 대시보드 진입 → 첫 참여
```

**Pain Points:**

| 단계 | 문제 | 근거 |
|------|------|------|
| 가입 신청 | "승인까지 1~2일 소요" 안내 후 아무런 후속 커뮤니케이션 없음 | `signup/page.tsx` — 완료 화면에서 로그인 버튼만 제공, 이메일 알림 없음 |
| 승인 대기 | 승인 상태를 확인할 방법이 없음 (폴링/실시간 알림 부재) | 미승인 상태에서 로그인 시도 시 UX 불명확 |
| 온보딩 부재 | 승인 후 첫 로그인 시 바로 대시보드로 이동 — 학회 소개, 활동 안내 없음 | `dashboard/page.tsx` — 환영 메시지는 있으나 온보딩 플로우 전무 |
| 기수(generation) 정보 | 가입 시 기수를 입력할 수 없음, 관리자가 수동 배정 | `SignupForm.tsx` — generation 필드 없음 |

### 2.2 기존 회원 여정

```
로그인 → 대시보드 → 세미나 신청 → 참석 → 후기 작성 → 게시판 활동
```

**Pain Points:**

| 단계 | 문제 | 근거 |
|------|------|------|
| 세미나 신청 | 내부 attendeeIds 방식과 외부 SeminarRegistrationForm이 혼용됨 — 어느 방식으로 신청해야 하는지 불명확 | `seminars/[id]/page.tsx` — `toggleAttendance` + `RegistrationSection` 동시 존재 |
| 세미나 공간 접근 | "세미나 공간 입장" 버튼이 등장하는 조건(attendeeIds + registrations 중 하나만 충족해도 표시)이 복잡, 미등록자도 LMS 접근 가능 | `SeminarLMS.tsx` — `hasAccess`가 `isAttending || isStaff`이나 LMS 자체는 미접근자에게도 탭 표시 |
| 게시판 네비게이션 | 5개 카테고리를 별도 URL로 운영(`/board/free`, `/board/promotion` 등)하지만 전체 게시판(`/board`)에서 카테고리 필터가 없음 | `board/page.tsx` — 검색만 있고 카테고리 탭 없음 |
| 활동 피드 | `ActivityFeed`가 내 게시글 기반으로만 동작 — 동기 회원들의 최근 활동을 볼 수 없음 | `dashboard/page.tsx` — `ActivityFeed userId={user.id}` |
| 학회보 | URL 기반 딥링크 없음 — 특정 호를 공유하거나 북마크 불가 | `newsletter/page.tsx` — state로 selectedId 관리, URL 변경 없음 |

### 2.3 졸업생 여정

```
오랜만에 방문 → 최근 소식 확인 → 후배 활동 확인 → (선택적) 참여
```

**Pain Points:**

| 단계 | 문제 | 근거 |
|------|------|------|
| 진입점 불명확 | 홈페이지가 로그인 여부와 무관하게 동일 — 졸업생 전용 콘텐츠나 환영 메시지 없음 | `page.tsx` — 모든 방문자에게 동일한 섹션 표시 |
| 네트워킹 부재 | 졸업생끼리 또는 재학생-졸업생 간 연결 기능 없음 | 구성원 페이지에 졸업생 탭 있으나 상호작용 불가 |
| 관심 세미나 알림 | 이메일/알림으로 세미나 소식 받을 방법 없음 | 알림 구독 기능 전무 |

### 2.4 운영진 여정

```
세미나 기획 → 홍보 콘텐츠 생성 → 신청자 관리 → 당일 QR 체크인 → 결과 리포트
```

**Pain Points:**

| 단계 | 문제 | 근거 |
|------|------|------|
| seminar-admin 진입 | `/seminar-admin`이 별도 경로이나, 어떤 세미나를 관리하는지 진입 전에 선택하지 않음 — 항상 최신 세미나를 기본값으로 가정 | `seminar-admin/page.tsx` |
| 포스터 생성 | AI 포스터 생성 결과물을 Firestore에 저장하지 않으면 재생성 필요 | `seminar-admin/poster/page.tsx` |
| 신청자 관리 | 등록 폼 응답 데이터를 CSV로 다운로드하는 기능 없음 | `RegistrationsTab.tsx` 확인 필요 |
| 수료증 발급 | 수료증 대량 발급 시 개별 다운로드만 가능 (일괄 ZIP 불가 추정) | `CertificateGenerator.tsx` |
| 타임라인 | 세미나 타임라인이 단독 `/seminar-admin/timeline` 경로 — 세미나 상세 페이지에서 바로 연결 없음 | 경로 구조 |

### 2.5 관리자 여정

**Pain Points:**

| 단계 | 문제 | 근거 |
|------|------|------|
| 회원 승인 | 승인 대기 회원에게 승인/거절 이메일 자동 발송 없음 | `AdminMemberTab.tsx` — 역할 변경만 있음 |
| 기수 배정 | 신규 회원 승인 시 기수를 수동으로 입력해야 함 — 자동 배정 로직 없음 | `AdminUserList.tsx` 확인 필요 |
| 사이트 설정 | 여러 설정 페이지가 별도 경로로 분산 — 한 화면에서 전체 개요 확인 불가 | `/admin/settings/*` 경로 다수 |

## 3. 기능 갭 분석

### 3.1 커뮤니케이션 & 알림 시스템 (완전 부재)

현재 사이트에는 이메일 알림, 인앱 알림, 푸시 알림이 전혀 없다.

| 필요 알림 | 수신자 | 현재 상태 |
|----------|--------|----------|
| 가입 신청 접수 확인 | 신청자 | 없음 |
| 회원 승인/거절 | 신청자 | 없음 |
| 새 세미나 등록 | 전체 회원 | 없음 |
| 세미나 D-1 리마인더 | 신청자 | 없음 |
| 댓글 알림 | 게시글 작성자 | 없음 |
| 관리자 승인 대기 | 운영진 | 없음 |

### 3.2 검색 기능 (부분 구현)

- 세미나 검색: 제목/발표자/장소 — 구현됨
- 게시판 검색: 제목/작성자 — 구현됨
- **전체 통합 검색**: 없음
- 학회보 검색: 없음
- 회원 검색 (공개 범위): 구성원 페이지에 검색 없음

### 3.3 개인화 & 추천

- 내 관심 분야(`field`) 기반 세미나 추천: 없음
- 최근 본 게시글: 없음
- 맞춤 홈피드: 없음

### 3.4 소셜 & 네트워킹

- 회원 간 DM/메시지: 없음
- 스터디 그룹 모집 게시판: 활동 페이지가 정적 텍스트만 있음
- 기수별 커뮤니티 공간: 없음

### 3.5 데이터 & 통계

- 세미나 참석 이력 시각화: 없음
- 회원 활동 통계 대시보드 (관리자용): 없음
- 게시판 조회수: 없음 (구현 여부 불명확)
- 학회 성장 지표 (회원 수 추이 등): 없음

### 3.6 모바일 UX

- PWA 지원: 없음
- 모바일 최적화: Tailwind 반응형은 있으나 체크인 QR 스캐너 등 모바일 핵심 기능 UX 미검증

### 3.7 접근성

- 키보드 네비게이션: shadcn/ui 기본 수준
- 스크린리더 지원: aria-label 일부만 적용
- 다크모드: 없음

## 4. UX 개선점

### 4.1 정보 아키텍처 문제

**문제: seminar-admin vs admin/seminars 경로 혼재**
- `/seminar-admin/` (운영진 세미나 운영 도구)
- `/admin/seminars` (관리자 세미나 목록)
- 두 경로의 역할 구분이 사용자에게 불명확

**문제: 게시판 카테고리 접근 방식 이중화**
- `/board` — 전체 게시글, 카테고리 필터 없음
- `/board/free`, `/board/seminar` 등 — 카테고리별 별도 URL
- 사용자가 어디서 글을 써야 하는지 혼란스러울 수 있음

### 4.2 빈 상태(Empty State) 처리

- 세미나 없을 때: 적절한 CTA(세미나 신청하기 등) 없이 단순 텍스트
- 게시글 없을 때: 동일
- 활동 피드가 비어있을 때: 명확한 안내 없음

### 4.3 로딩 & 에러 처리

- Skeleton UI: 세미나 목록에만 구현, 나머지 페이지는 스피너 또는 없음
- 에러 경계: `error.tsx` 존재하나 세부 에러 유형 구분 없음
- 오프라인 상태 처리: 없음

### 4.4 세미나 등록 플로우 불명확성

세미나 상세 페이지에 두 가지 참석 방식이 공존한다:
1. `toggleAttendance` — attendeeIds 배열에 userId 추가 (단순 버튼)
2. `SeminarRegistrationForm` — 외부 폼 또는 커스텀 필드 입력

사용자 입장에서 둘의 차이가 명확하지 않으며, LMS 접근 권한 판단에도 두 방식이 모두 사용된다.

### 4.5 학회보 딥링크 부재

학회보 특정 호를 외부에 공유하거나 직접 URL로 접근할 수 없다. state 기반 네비게이션으로 인해 `newsletter/page.tsx`가 항상 목록을 보여주고, 호 선택은 클라이언트 상태로만 관리된다. 이는 SEO와 공유성을 저하시킨다.

### 4.6 AI 챗봇 진입점 인지성

AI 챗봇(ChatWidget)이 우하단 FAB로만 존재한다. 회원들이 이 기능을 인지하지 못할 가능성이 높고, "AI 학회장"이라는 명칭이 처음 방문자에게 무슨 기능인지 전달하지 못한다.

## 5. 성장 및 참여 전략

### 5.1 회원 활성화 전략

| 전략 | 설명 | 기대 효과 |
|------|------|----------|
| 온보딩 플로우 | 최초 로그인 시 학회 소개 → 관심 분야 설정 → 다음 세미나 안내 3단계 | 신규 회원 이탈률 감소 |
| 이메일 알림 | 세미나 리마인더, 댓글 알림, 게시글 좋아요 | 재방문율 향상 |
| 참여 이력 시각화 | 마이페이지에 "나의 참여 통계" (세미나 N회, 게시글 N개, 댓글 N개) | 기여감 강화 |
| 기수별 커뮤니티 | 같은 기수 회원들의 활동 모아보기 | 소속감 강화 |

### 5.2 재방문율 향상

| 전략 | 설명 | 기대 효과 |
|------|------|----------|
| 대시보드 개인화 | 관심 분야 기반 추천 세미나, 기수 동료 최근 활동 | 대시보드 방문 동기 부여 |
| 주간 다이제스트 이메일 | 지난 주 세미나 요약, 새 게시글 Top 3, 다음 세미나 | 주기적 재방문 유도 |
| 세미나 후기 문화 | 세미나 완료 후 참석자에게 후기 작성 넛지 | 콘텐츠 누적 가속 |

### 5.3 외부 유입 (잠재 회원/관심자)

| 전략 | 설명 | 기대 효과 |
|------|------|----------|
| 세미나 공개 아카이브 | 완료된 세미나의 발표자/주제를 비회원에게 공개 | 학회 브랜딩, 가입 전환 |
| 학회보 공개 배포 | 학회보 PDF/웹 버전을 공개 링크로 공유 가능 | 학회 가시성 확대 |
| 연사 프로필 강화 | 게스트 연사 소개 페이지를 별도로 구성 (GuestSpeakersSection 확장) | SEO, 학회 신뢰도 |

## 6. MoSCoW 우선순위 제안

### Must (즉시 필요 — 핵심 운영 기능)

| ID | 기능 | 근거 |
|----|------|------|
| M-01 | 회원 승인 이메일 알림 | 현재 승인 후 회원이 인지하는 방법 없음 — 운영 필수 |
| M-02 | 세미나 등록 방식 단일화 | `toggleAttendance` vs `RegistrationForm` 혼용 — 데이터 일관성 문제 |
| M-03 | 가입 신청 → 승인 상태 조회 | 신청 완료 후 상태 확인 불가 — 신규 회원 첫 경험 |
| M-04 | 학회보 URL 딥링크 | 현재 공유/북마크 불가 — SEO 및 기본 사용성 |
| M-05 | 신규 회원 기수 가입 시 입력 필드 추가 | 관리자 수동 배정 부담 해소 |

### Should (중요 — 회원 경험 향상)

| ID | 기능 | 근거 |
|----|------|------|
| S-01 | 최초 로그인 온보딩 플로우 | 신규 회원 이탈 방지 — 임팩트 높음 |
| S-02 | 세미나 D-1 이메일 리마인더 | 세미나 참석률 향상 |
| S-03 | 게시판 전체 뷰에 카테고리 탭 | 현재 `/board` — 필터 없음 |
| S-04 | 신청자 목록 CSV 다운로드 | 운영진 행정 업무 경감 |

## 11. Architecture Considerations

### 11.1 Project Level

| Level | Selected | Rationale |
|-------|:--------:|-----------|
| Dynamic | O | 현재 구조 — `src/features/`, `src/components/`, BaaS(bkend.ai) 기반 |

### 11.2 이메일 알림 도입 시 아키텍처

```
옵션 A: Resend API (권장)
  - Next.js API Route → Resend SDK → 이메일 발송
  - 환경변수: RESEND_API_KEY

옵션 B: Firebase Extensions (Trigger Email)
  - Firestore 문서 작성 → 자동 이메일 발송
  - Firebase Console에서 확장 설치

비용: Resend 무료 플랜 100건/일, Firebase Extensions 별도 SMTP 설정 필요
```

### 11.3 학회보 URL 구조 변경

```
현재: /newsletter (state 기반)
변경: /newsletter → 목록 (유지)
      /newsletter/[id] → 개별 호 (신규)
```
